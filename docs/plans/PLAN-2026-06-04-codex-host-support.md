# PLAN ‚Äî Codex host support

**Date:** 2026-06-04
**Status:** Draft (research-grounded; Phase 0 verification gate before build)
**Owner:** great_cto core
**Related:** `adapt.ts` (cross-AI config gen), ADR-018/019 (assumed Codex = manual DCDSV ‚Äî now outdated), canary.sh (multi-platform dropped at v2.8.5)

## Goal

Make **OpenAI Codex CLI** a first-class great_cto host alongside Claude Code ‚Äî
the hook chain, the 57 agents, the commands, the MCP server, and the AGENTS.md
brain all work under Codex, with an explicit, documented degradation list for
anything that genuinely can't map.

## Why now

great_cto **had** multi-platform support (codex/cursor/aider/continue) and
**dropped it at v2.8.5** to "Claude Code-only" ‚Äî correctly, because Codex then
lacked the primitives. **Codex 2026 has closed the gap.** Verified against
`developers.openai.com/codex/config-reference`, Codex now supports:

- **Hooks** ‚Äî `[hooks.<Event>]` (or `hooks.json`), command hooks with matcher
  groups, behind `[features].hooks`. Events: PreToolUse, PermissionRequest,
  PostToolUse, PreCompact, PostCompact, **SessionStart**, **SubagentStart**,
  SubagentStop, UserPromptSubmit, Stop. ‚Üí near-identical to great_cto's CC hook set.
- **Subagents** ‚Äî `[agents.<name>]` with `config_file`, `description`,
  `max_threads`/`max_depth`/`job_max_runtime_seconds`.
- **MCP** ‚Äî `[mcp_servers.<id>]` (command/args/cwd/env + approval controls).
- **AGENTS.md** ‚Äî discovered + concatenated (root‚Üífolder), `project_doc_fallback_filenames`.
- Config layering: `~/.codex/config.toml` (global) + `.codex/config.toml` (project, trusted).

## Primitive mapping (the core of this plan)

| great_cto primitive | Delivery on Claude Code | Delivery on Codex |
|---|---|---|
| Hook chain (`scripts/hooks/*.mjs`) | `plugin.json` `hooks{}` | `~/.codex/hooks.json` (or `[hooks]` in config.toml) + `[features].hooks=true`. **Same .mjs scripts** ‚Äî they're host-agnostic Node, read stdin JSON, signal via exit code. |
| 57 agents (`agents/*.md`) | copied to `~/.claude/agents/great_cto-*.md` | `[agents.<name>]` entries with `config_file` ‚Üí the agent body |
| Commands (`commands/*.md`) | copied to `~/.claude/commands/*.md` | custom prompts (slash) ‚Äî **format TBD (Phase 0)**; fallback: `[agents]` or AGENTS.md routing |
| MCP server (`mcp.ts`) | `.mcp.json` / plugin | `[mcp_servers.great_cto]` ‚Üí `command="npx", args=["great-cto","mcp"]` |
| Brain / classifier / gates | CLAUDE.md + SessionStart inject | **AGENTS.md** (already generated by `adapt.ts`) ‚Äî extend with classifier + routing table + gates |
| Companion plugins (superpowers, Beads) | auto-installed | **degraded** ‚Äî superpowers/Beads are Claude-Code plugins; agents must keep the existing `HOST=claude-code` guards + fallback notes |

## Critical unknowns ‚Äî Phase 0 verification gate (do FIRST, no build before)

The config reference confirms the *structure* but not these, which the install
path depends on. Verify against a **real Codex install** before writing code:

1. **Hook stdin payload shape.** great_cto's `.mjs` parse Claude Code's shape
   (`tool_name`, `tool_input`, `tool_response`, `is_error`). Does Codex send the
   same keys? If not, hooks need a payload-normalization shim.
2. **Hook block/deny signal.** CC uses exit 2 + `hookSpecificOutput.permissionDecision`.
   Does Codex honor exit 2? Same `permissionDecision` JSON? (secret-scan depends on this.)
3. **Custom-prompt format + dir** for mapping commands ‚Üí Codex slash commands.
4. **Plugin path resolution** ‚Äî CC hooks resolve `${PLUGIN_DIR}` from
   `~/.claude/plugins/cache/...`. Codex install must write a resolvable path
   (npx global path or a `~/.great_cto/codex/` copy).
5. **`[features].hooks` stability** ‚Äî is it GA or experimental? Gate accordingly.

Phase 0 deliverable: a short findings doc + 1 working end-to-end hook on Codex
(e.g. tool-failure logging) proving the payload + path + signal.

## Phased plan

### Phase 1 ‚Äî `great-cto install --host codex` (or `adapt --platform codex`)
- Re-introduce `codex` to `adapt.ts` `AiTool`/`Platform` (it was removed; a
  vestigial `--platform codex` test still exists in test-pipeline.sh).
- Generate: `~/.codex/config.toml` fragment (`[mcp_servers.great_cto]`,
  `[features].hooks=true`, `[agents.*]`), `~/.codex/hooks.json`, project AGENTS.md.
- Idempotent + non-destructive (merge into existing config.toml, never clobber).

### Phase 2 ‚Äî host-agnostic hook scripts
- Audit `scripts/hooks/*.mjs` for Claude-Code assumptions (payload keys, plugin
  path). Add a thin payload-normalizer if Phase 0 shows divergence.
- Wire the 9 mapped events into Codex hooks.json.

### Phase 3 ‚Äî agents + commands translation
- Emit `[agents.<name>]` for the 57 agents (config_file ‚Üí agent body).
- Map commands ‚Üí Codex custom prompts (per Phase 0 format).
- Keep superpowers/Beads behind `HOST` guards (already present).

### Phase 4 ‚Äî degradation doc + tests
- `docs/HOSTS.md`: capability matrix CC vs Codex; what degrades and why.
- Tests: `adapt --platform codex` golden output; hooks.json schema; a Codex
  smoke run in test-pipeline.sh.

## What will NOT map (honest degradation)
- **superpowers / Beads plugins** ‚Äî Claude-Code-specific. Codex users get the
  fallback prose paths the agents already carry behind `HOST=claude-code`.
- **Exact gate UX** ‚Äî gates become AGENTS.md-documented manual checkpoints unless
  Codex hook semantics allow the same inline approval.
- Anything Phase 0 reveals as Codex-incompatible.

## Risks
- **Codex evolving fast** ‚Äî schemas may shift; pin to a Codex version in tests,
  re-verify on bumps. (Same posture as ADR-018 for workflows.)
- **Two-host maintenance** ‚Äî every new hook/agent must ship both wirings.
  Mitigate: single source (`.mjs` scripts + agent .md), generated per-host.
- **`[features].hooks` may be experimental** ‚Äî if so, mark Codex hook support beta.
- **Privacy** ‚Äî Codex config in `~/.codex/`; keep the zero-telemetry + no-private-
  -names rules; AGENTS.md must not embed private project names.
