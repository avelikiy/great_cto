# RESCORE 2026-07-19 — the executing oracle is not ready to replace the filename oracle

**Task:** T1 of [PLAN-2026-07-19](../plans/PLAN-2026-07-19-harness-close-the-loops.md) —
re-score all 10 benchmark products through the *executing* oracle
(`scripts/lib/quality.mjs` → `scripts/lib/product-eval.mjs`) and compare against the
published filename-based scores, to size the drop before deciding whether to republish.

**Outcome: do not republish. The re-scored numbers are an artifact of a broken test
parser, not a measurement of product quality.** The measurement did its job — it caught
this *before* the oracle was wired into `bench-collect.mjs` and before a corrected
benchmark was published on numbers that were themselves wrong.

**Cost:** $0. No source file was modified to produce this.

## Why the numbers are invalid

### Defect 1 — the executing oracle cannot read the test runner every product uses

`parseTestCounts()` (`scripts/lib/product-eval.mjs:63`) matches only the
**node:test / TAP** summary shape:

```
^[#ℹ]\s*tests\s+(\d+)      →  "# tests 269"   ✅
```

All ten benchmark products run **vitest** (`"test": "vitest run"`), whose summary is
`Tests  196 passed (196)`. Verified directly:

| Input | `parseTestCounts` |
|-------|-------------------|
| real vitest summary | `{total: null, pass: null, fail: null}` |
| real node:test summary | `{total: 269, pass: 269, fail: 0}` |

So for every product under test the parser extracts nothing.

### Defect 2 — a parse failure is silently converted into a failed test

With counts unparsed, `runEval()` falls back:

```js
const total = c.total ?? 0, fail = c.fail ?? (r.status === 0 ? 0 : 1);
tests = { ran: true, total: total || (r.status === 0 ? 1 : 1), passed: (total || 1) - fail, failed: fail };
```

`ran: true` is reported even though nothing was actually counted, and the product is
scored as **"1 test, and it failed."** Nine of ten products came back `tests: 0/1 pass` —
including `ats` and `booking`, which the benchmark's own collector measured at **269/269**
and **368/368**.

This is the same silent-failure pattern as the board readers (F2): *the component could
not do its job, and reported a confident wrong answer instead of saying so.* Here it is
worse than a blank panel, because the wrong answer is a published quality score.

### Confound 3 — the re-run had no test environment

`portal` and `dispatch` need a live Postgres (their suites provision an ephemeral schema);
Docker was not running at re-score time. Their suites would fail now regardless of the
parser. Any honest re-score must provision each product's environment first — otherwise
"environment did not come up" is recorded as "product is bad", which is exactly the
inverted version of the defect this task set out to fix.

## What the two oracles actually disagree about

The repository already contains a **working** test parser: `bench-collect.mjs`'s own
`runTests()` produced the published `269/269`, `368/368`, `196/196` counts, and correctly
recorded `coaching` as `exit 143` with no counts. The problem is not that the executing
oracle is stricter — it is that the executing oracle's parser is blind while the
collector's is not.

| | `bench-collect.runTests()` | `product-eval.parseTestCounts()` |
|--|---------------------------|----------------------------------|
| vitest | ✅ parses | ❌ returns nulls |
| node:test | ✅ parses | ✅ parses |
| suite never ran | records `exit 143`, no counts | reports `ran: true`, `0/1` |

## Consequences for the plan

**T5 and T6 must not proceed as written.** Wiring `quality.mjs` into the collector today
would replace a scorer that over-credits (filename globbing) with one that under-credits
for a mechanical reason (parser blindness) — trading a known-weak oracle for a broken one,
while making the benchmark *look* more rigorous.

Revised prerequisites, in order:

1. **T5a (new, blocking):** teach `parseTestCounts` the runners actually in use — vitest,
   jest, and the node:test shape it already handles — with a fixture test per format.
   Fixtures should be captured from real product output, not hand-written.
2. **T5b (new, blocking):** when counts cannot be parsed, `tests.ran` must be `false` and
   the score `null`. "I could not measure" must never render as "measured: failed". This
   is the same rule T5 already states for a suite that never ran, applied to the parse
   path.
3. **T1-redo:** re-score with the environment provisioned (the Postgres clusters the
   collection run used), and only then compare rankings against the human judgements in
   the BENCH row details.
4. Only then T6 (wire the collector) and T7 (devops gate).

**The published table stands unchanged for now.** Its methodology section already labels
`score` a structural floor rather than proof of correctness, and marks the three
interrupted runs as not comparable — which remains the accurate description.

## Second run — parser fixed, environment still not

After the parser learned vitest/jest and the null rule landed, the re-score was
repeated with the two Postgres clusters the collection run had used.

| Product | Published | Re-scored | tests read | floor | ceiling |
|---------|-----------|-----------|-----------|-------|---------|
| dashboard | 86 | 85 (B) | **157/157** | 86 | 85 |
| booking | 58 | **81 (B)** | 320/368 | 58 | 94 |
| subs | 70 | 79 (B) | **54/54** | 70 | 85 |
| ats | 74 | 72 (C) | 97/269 | 74 | 70 |
| dispatch | 66 | 68 (C) | 147/226 | 66 | 69 |
| leadcrm | 80 | 68 (C) | 62/132 | 80 | 61 |
| quoting | 53 | 67 (C) | 366/553 | 53 | 75 |
| portal | 63 | 66 (C) | 56/196 | 63 | 68 |
| classes | 70 | **null** | not measured | 70 | 45 |
| coaching | 76 | **null** | not measured | 76 | 55 |

**What is now working.** The parser reads real counts everywhere (269, 368, 553…)
instead of the previous `0/1`. And the null rule fires exactly where it should:
`coaching` — the product whose suite exited 143 and was nonetheless published as
**76 (B)** — now carries no grade at all, which was the whole point.

The movement is also two-directional, which the filename oracle could never show:
`booking` rises 58 → 81 because its suite genuinely passes and its typecheck,
lint and audit are clean, while `leadcrm` falls 80 → 68 because its structural
floor was flattering a suite that does not pass.

**Why it is still not publishable.** Compare against what the collection run
measured with each product's own environment up:

| Product | at collection | at re-score | |
|---------|--------------|-------------|--|
| ats | 269/269 | 97/269 | −172 |
| portal | 196/196 | 56/196 | −140 |
| dispatch | 225/226 | 147/226 | −78 |
| leadcrm | 130/132 | 62/132 | −68 |

Only `dashboard` and `subs` run green, and they are the two products that need no
external services. The rest are measuring **absent infrastructure**, not code. Two
generic Postgres clusters are not the same as each product's own compose stack —
`ats` alone lost 172 tests to services nobody started.

**Conclusion.** The oracle is now honest about what it can and cannot see, which
is the prerequisite that was missing. A publishable re-score additionally needs
per-product environment provisioning — bringing up each product's declared
services before its suite runs. Until then the published table stands, and T6
(wiring the collector) stays blocked: swapping in an oracle that measures the CI
host's missing services would be a different way of publishing the wrong number.

## Third run — environments provisioned per product (bench-env)

With `scripts/bench-env.mjs` bringing up each product's declared database
(relocated off the unadministerable system 5432, schemas migrated with each
product's own script), the re-score becomes meaningful for 7 of 10:

| Product | Published | Re-scored (env up) | tests read |
|---------|-----------|--------------------|-----------|
| portal | 63 | **86 (B)** | **196/196** |
| dashboard | 86 | 85 (B) | **157/157** |
| leadcrm | 80 | 83 (B) | 131/132 |
| booking | 58 | **81 (B)** | 320/368 |
| classes | 70 | 81 (B) | 302/316 |
| subs | 70 | 79 (B) | **54/54** |
| dispatch | 66 | 78 (B) | 225/226 |
| ats | 74 | 72 (C)¹ | 97/269¹ |
| quoting | 53 | 67 (C)² | 366/553² |
| coaching | 76 | **null** | not measured³ |

¹ ats's migration script fails on this host (its own `db:migrate` crashes), so
its suite still runs against an unmigrated schema — its executed number remains
an environment artifact, not a product verdict.
² quoting was stopped mid-build; its failures are a half-wired product and the
published table already marks it not-comparable.
³ coaching's suite exits without ever printing a summary (SIGTERM) — the null is
the oracle telling the truth.

**Reading.** Test counts now match collection-time almost exactly (portal
196/196, dispatch 225/226, leadcrm 131/132 vs 130/132). The executed oracle
moves products in both directions relative to the filename score — portal
63→86 and booking 58→81 rise because their suites genuinely pass; nothing
inflates. Median across the 7 cleanly-measured products: **81 (B)** vs the
published 70. The remaining blockers for publication are one product's own
broken migration script (ats) and the two interrupted builds — a much narrower
caveat than "the oracle can't read vitest".
