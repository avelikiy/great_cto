# Reliability ‚Äî SLO + Error Budgets ‚Äî Reference

> **SLO defines the target. INCIDENT-LOG records reality. Budget is the gap.** When budget is burned, feature velocity automatically throttles ‚Äî no "should we stabilize?" debate.

## Files

### `docs/reliability/SLO.md`

Single file. Per-service SLO definitions + response policy. Owner: tech-lead creates during ARCH, CTO edits manually.

```markdown
# SLO ‚Äî <project>

> Per-service Service Level Objectives. See `skills/great_cto/references/reliability.md`.

## Services

### api
| SLI | Target | Budget (30d rolling) | Window |
|-----|--------|----------------------|--------|
| Availability (HTTP 2xx / total) | 99.9% | 43.2 min downtime | 30d rolling |
| Latency p95 | < 200ms | 2h over threshold | 30d rolling |
| Error rate | < 0.5% | 3.6h > threshold | 30d rolling |

### billing
| SLI | Target | Budget (30d) | Window |
|-----|--------|--------------|--------|
| Availability | 99.95% | 21.6 min | 30d rolling |
| Charge success rate | > 99.9% | 43.2 min (of charging time) | 30d rolling |

## Response policy
- **Budget < 50%** ‚Äî notify #engineering (informational)
- **Budget < 20%** ‚Äî warn at gate:ship, devops flags risky deploys
- **Budget < 0%** (budget exceeded) ‚Äî freeze feature deploys until next window recovers; hotfix-only mode
- **Multiple SLIs burned simultaneously** ‚Äî tech-lead drafts stability plan, CTO approves

## SLO change procedure
Changing an SLO target (e.g. 99.9% ‚Üí 99.95%) requires ADR.
Tightening the target retroactively backfills the INCIDENT-LOG impact calculations.
```

### `.great_cto/slo-budget-current.md` (computed cache)

Regenerated by `/digest`. `/inbox` reads it without recompute.

```markdown
# SLO Budget ‚Äî current state (updated 2026-04-19)

| Service | SLI | Window | Used | Remaining | Status |
|---------|-----|--------|------|-----------|--------|
| api | availability | 30d rolling | 15min (35%) | 28.2min | ok |
| api | latency p95 | 30d rolling | 8min (7%) | 112min | ok |
| billing | availability | 30d rolling | 18.4min (85%) | 3.2min | WARN |
```

Status thresholds: `ok` (< 50% used) / `warn` (50-80%) / `WARN` (80-100%) / `EXHAUSTED` (> 100%).

## Incident entry format ‚Äî exact fields

Each entry is a level-2 heading followed by exactly 4 lines:

```
## <ISO-8601-UTC> | <service> | <short title>
Cause: <one-liner root cause>
SLI impact: <which SLI, magnitude, and window delta>
Budget consumed: <min consumed> of <window budget> (<percent>)
Postmortem: <PM-id or dash>
```

Keep lines terse ‚Äî they're grep'd by `/digest` to compute budgets. No freeform text between entries.

## Budget computation (bash-only, no dependencies)

```bash
# Example: 30d rolling availability budget for "api"
CUTOFF=$(date -v-30d +%Y-%m-%dT%H:%M 2>/dev/null || date -d "30 days ago" +%Y-%m-%dT%H:%M)
awk -v cut="$CUTOFF" '
  /^## [0-9]{4}-/ { d=$2; service=$4; next }
  service == "api" && /Budget consumed:/ && d >= cut {
    match($0, /[0-9]+/); print substr($0, RSTART, RLENGTH)
  }
' docs/reliability/INCIDENT-LOG.md | awk '{sum+=$1} END {print sum}'
```

## Enforcement in the pipeline

### gate:ship budget check (devops)

Before promoting canary ‚Üí full rollout, devops reads `.great_cto/slo-budget-current.md` for the service being deployed:

- All SLIs at `ok` ‚Üí proceed normally
- Any SLI at `warn` ‚Üí log in RELEASE doc, proceed
- Any SLI at `WARN` (80%+) ‚Üí WARN CTO in chat, proceed only after explicit approval
- Any SLI at `EXHAUSTED` (>100%) ‚Üí BLOCK. Only CTO override via WAIVER (see `waivers.md`) permits deploy. Hotfix exception: yes if incident ID present linking to outage we're fixing.

### Policy-change ceremony

When budget hits `EXHAUSTED`, the recovery is not "wait for window to roll over." tech-lead drafts a stability plan (`docs/reliability/STABILITY-PLAN-<date>.md`) within 24h; CTO reviews; team executes for a week before new features resume.

## Consumers

- `tech-lead` ‚Äî seeds SLO.md entry when introducing new service in ARCH
- `l3-support` ‚Äî appends to INCIDENT-LOG on every postmortem
- `devops` ‚Äî appends on canary failure; blocks deploys on EXHAUSTED budget
- `/inbox` ‚Äî shows SLIs at WARN/EXHAUSTED
- `/digest` ‚Äî recomputes `slo-budget-current.md`; weekly trend; quarterly burn-down
- Q-review (v1.0.75) ‚Äî budget-burn summary + policy-change frequency

## Not in scope (deliberately)

- Real-time monitoring integration (Datadog/Prometheus) ‚Äî out of scope, our mode is **manual reality log + computed budget**
- Per-endpoint SLOs ‚Äî granularity ends at service
- Automated alerting ‚Äî alerts stay in your existing monitoring system; this is a **decision record**, not a paging system
