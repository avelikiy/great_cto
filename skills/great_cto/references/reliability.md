# Reliability — SLO + Error Budgets — Reference

> **SLO defines the target. INCIDENT-LOG records reality. Budget is the gap.** When budget is burned, feature velocity automatically throttles — no "should we stabilize?" debate.

## Files

### `docs/reliability/SLO.md`

Single file. Per-service SLO definitions + response policy. Owner: tech-lead creates during ARCH, CTO edits manually.

```markdown
# SLO — <project>

> Per-service Service Level Objectives. See `skills/great_cto/references/reliability.md`.

## Services

### api
| SLI | Target | Budget (30d rolling) | Window |
|-----|--------|----------------------|--------|
| Availability (HTTP 2xx / total) | 99.9% | 43.2 min downtime | 30d rolling |
| Latency p95 | < 200ms | 2h over threshold | 30d rolling |
| Error rate | < 0.5% | 3.6h > threshold | 30d rolling |

### billing
| SLI | Target | Budget (30d) | Window |
|-----|--------|--------------|--------|
| Availability | 99.95% | 21.6 min | 30d rolling |
| Charge success rate | > 99.9% | 43.2 min (of charging time) | 30d rolling |

## Response policy
- **Budget < 50%** — notify #engineering (informational)
- **Budget < 20%** — warn at gate:ship, devops flags risky deploys
- **Budget < 0%** (budget exceeded) — freeze feature deploys until next window recovers; hotfix-only mode
- **Multiple SLIs burned simultaneously** — tech-lead drafts stability plan, CTO approves

## SLO change procedure
Changing an SLO target (e.g. 99.9% → 99.95%) requires ADR.
Tightening the target retroactively backfills the INCIDENT-LOG impact calculations.
```

### `docs/reliability/INCIDENT-LOG.md` (append-only)

One file, reverse-chronological (newest on top for quick scan) or chronological (easier to compute window). **Chronological** (newest at bottom) — makes windowing with standard unix tools easy.

```markdown
# Incident Log — append only

> Every incident that impacts an SLI. Updated by l3-support (on postmortem) + devops (on canary failure) + CTO (manual).
> See `skills/great_cto/references/reliability.md` for format.

## 2026-04-18T14:22Z | api | 15min downtime
Cause: DB connection pool exhaustion during traffic spike
SLI impact: availability -0.035% of window | latency p95 not affected
Budget consumed: 15min of 43.2min (35%)
Postmortem: PM-003

## 2026-04-19T09:15Z | api | p95 spiked 400ms for 8min
Cause: slow N+1 query introduced in #PR-452
SLI impact: latency p95 breach 8min
Budget consumed: 8min of 120min (7%)
Postmortem: —
```

### `.great_cto/slo-budget-current.md` (computed cache)

Regenerated by `/digest`. `/inbox` reads it without recompute.

```markdown
# SLO Budget — current state (updated 2026-04-19)

| Service | SLI | Window | Used | Remaining | Status |
|---------|-----|--------|------|-----------|--------|
| api | availability | 30d rolling | 15min (35%) | 28.2min | ok |
| api | latency p95 | 30d rolling | 8min (7%) | 112min | ok |
| billing | availability | 30d rolling | 18.4min (85%) | 3.2min | WARN |
```

Status thresholds: `ok` (< 50% used) / `warn` (50-80%) / `WARN` (80-100%) / `EXHAUSTED` (> 100%).

## Incident entry format — exact fields

Each entry is a level-2 heading followed by exactly 4 lines:

```
## <ISO-8601-UTC> | <service> | <short title>
Cause: <one-liner root cause>
SLI impact: <which SLI, magnitude, and window delta>
Budget consumed: <min consumed> of <window budget> (<percent>)
Postmortem: <PM-id or dash>
```

Keep lines terse — they're grep'd by `/digest` to compute budgets. No freeform text between entries.

## Budget computation (bash-only, no dependencies)

```bash
# Example: 30d rolling availability budget for "api"
CUTOFF=$(date -v-30d +%Y-%m-%dT%H:%M 2>/dev/null || date -d "30 days ago" +%Y-%m-%dT%H:%M)
awk -v cut="$CUTOFF" '
  /^## [0-9]{4}-/ { d=$2; service=$4; next }
  service == "api" && /Budget consumed:/ && d >= cut {
    match($0, /[0-9]+/); print substr($0, RSTART, RLENGTH)
  }
' docs/reliability/INCIDENT-LOG.md | awk '{sum+=$1} END {print sum}'
```

## Sources — who writes incidents

| Trigger | Who | Postmortem link? |
|---------|-----|------------------|
| P0/P1 incident → postmortem generated | l3-support | yes (PM-NNN) |
| Canary health trip during deploy | devops | — (or create PM if severity warrants) |
| Performance regression caught post-deploy | l3-support or devops | — |
| Third-party vendor outage impacting us | CTO manual or l3-support | — |
| Silent SLI breach detected from monitoring | CTO manual | — |

## Enforcement in the pipeline

### gate:ship budget check (devops)

Before promoting canary → full rollout, devops reads `.great_cto/slo-budget-current.md` for the service being deployed:

- All SLIs at `ok` → proceed normally
- Any SLI at `warn` → log in RELEASE doc, proceed
- Any SLI at `WARN` (80%+) → WARN CTO in chat, proceed only after explicit approval
- Any SLI at `EXHAUSTED` (>100%) → BLOCK. Only CTO override via WAIVER (see `waivers.md`) permits deploy. Hotfix exception: yes if incident ID present linking to outage we're fixing.

### Policy-change ceremony

When budget hits `EXHAUSTED`, the recovery is not "wait for window to roll over." tech-lead drafts a stability plan (`docs/reliability/STABILITY-PLAN-<date>.md`) within 24h; CTO reviews; team executes for a week before new features resume.

## Consumers

- `tech-lead` — seeds SLO.md entry when introducing new service in ARCH
- `l3-support` — appends to INCIDENT-LOG on every postmortem
- `devops` — appends on canary failure; blocks deploys on EXHAUSTED budget
- `/inbox` — shows SLIs at WARN/EXHAUSTED
- `/digest` — recomputes `slo-budget-current.md`; weekly trend; quarterly burn-down
- Q-review (v1.0.75) — budget-burn summary + policy-change frequency

## Not in scope (deliberately)

- Real-time monitoring integration (Datadog/Prometheus) — out of scope, our mode is **manual reality log + computed budget**
- Per-endpoint SLOs — granularity ends at service
- Automated alerting — alerts stay in your existing monitoring system; this is a **decision record**, not a paging system
