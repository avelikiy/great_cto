## Services are the new software

The next wave isn't tools for specialists вҖ” it's **autopilots that sell the outcome of a service**.
An autopilot runs a whole business function end to end (intake вҶ’ process вҶ’ decide вҶ’ deliver) and
escalates only the judgment calls to a qualified human. Every model improvement makes the service
faster and cheaper.

GreatCTO ships those autopilots вҖ” each one a **flow of agents + tools with a human on the risky
steps**, a built-in compliance reviewer, and **live connectors** that run each flow on real data.

## Under the hood (for the CTO who runs it)

Each autopilot is built and operated by a gated pipeline of specialist agents вҖ” architect, 12-angle
reviewer, QA, security officer, devops вҖ” tuned to your stack and jurisdiction. **You make two
decisions per feature; everything else runs automatically.** Every vertical ships with its **own
domain compliance reviewer** вҖ” False Claims Act & NCCI for coding, OFAC & BSA for AML, FDCPA & Reg F
for collections, 21 CFR Part 11 for trials, ECOA for lending, ALTA for title, FMCSA for freight вҖ”
that blocks an unsafe design before it ever runs. Each reviewer is held to an **adversarial golden
test set** in CI before release. The reviewers, signed human gates, audit trail, and live connectors
are the trust layer that makes it safe to let the autopilot run.

**The permission is never the wound.** Every flow step is tagged reversible or not; the runtime
**refuses to execute an irreversible action autonomously** вҖ” money moves, claim submission, e-signing,
fleet changes and tax filing run *only* after a named human signs the checkpoint. Each autopilot also
declares an **accountable owner** вҖ” one person answers for what it does. `flow-runner.mjs <vertical>
--validate` enforces the invariant; all nineteen autopilots ship green.

## How it works

**`npx great-cto init`** вҖ” scans your stack and README, detects jurisdiction (GDPR? HIPAA? PCI?), writes `.great_cto/FLOW.md` with the exact agents, gates, and compliance frameworks for your project.

**`/start "describe the feature"`** вҖ” critics review the architecture and spec before any code is written. You review the plan at `gate:plan`.

**Agents run automatically** вҖ” senior-dev implements with TDD, 12-angle review, QA, security, devops. You approve ship at `gate:ship`.

## Two decisions per feature

```
рҹҹЎ gate:plan   вҶҗ  you decide here (architecture + tasks + cost)
   вҶ“
рҹӨ– senior-dev вҶ’ 12-angle review вҶ’ qa-engineer вҶ’ security-officer вҶ’ devops
   вҶ“
рҹҹў gate:ship   вҶҗ  you decide here (PR ready, security signed off)
```

Architects, planners, reviewers, QA, security, DevOps run automatically between those two human checkpoints. **Memory persists** between sessions: every gate verdict appends to `~/.great_cto/decisions.md`, every retrospective appends to per-project `lessons.md`, and `/crystallize` promotes high-impact patterns to a global library agents query before re-solving.

## Critics before the plan

The most expensive bugs aren't in the code вҖ” they're in decisions made before coding starts. Three critic agents run before the Plan stage, at the three positions where a mistake costs the most:

| Critic | Catches |
|---|---|
| **Architecture critic** | Coupling that rules out multi-tenancy later В· "obvious" O(nВІ) on real-scale data В· circular dependencies between bounded contexts |
| **Spec critic** | "We solved the wrong problem" вҖ” the worst class of bug, because no unit test will catch it В· misaligned acceptance criteria В· scope that was never agreed on |
| **Schema critic** | `NOT NULL` without a default on a 50M-row table (deadlock in 10min after deploy) В· missing `CONCURRENTLY` on index creation В· irreversible migrations with no rollback path |

Previously critics only activated starting from Plan. Now the pipeline catches architectural and spec-level mistakes before implementation begins вҖ” when reverting costs hours, not days.

## Limitations & non-goals

- **Not for teams** вҖ” solo-CTO is the product. 2+ engineers? You've outgrown it.
- **Not a replacement for senior engineers** вҖ” codifies process; doesn't make architectural judgement calls without one.
- **Not a CI/CD system** вҖ” gates run locally / in-session. You still need GitHub Actions for actual merge.
- **Not certification-audited** вҖ” PCI/HIPAA/SOC2 archetype scaffolds are starting points, not certifications.
- **Not deterministic** вҖ” LLM-generated outputs. Every gate verdict should be sanity-checked.
